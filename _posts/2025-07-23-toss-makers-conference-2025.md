## Data Driven 팀, 이렇게 시작됐습니다

- 데이터 분석은 **서비스에 대한 질문(=가설, 아젠다)을 중심**으로 시작
    - 하나의 아젠다 = 서비스에 던지는 하나의 질문

- 실제 아젠다 예시
    - "리워드 수령율의 임계점이 어떻게 될 것 같나요?"
    - "리워드 결과안내 푸시 변경 실험으로 재참여율은 얼마나 증가했나요?"
        
### 기존 프로세스의 한계점

- **휘발되는 분석 결과**
    - 분석 결과가 공유된 후 그대로 종료되는 패턴 반복
    - 인사이트들이 일회성으로 소비되고 재활용되지 못하는 상황
        
- **산재된 분석 파일**
    - 분석을 진행하고 나면 다양한 형태의 파일들이 생성되는데, 이들이 체계적으로 관리되지 않음
	- SQL 쿼리 파일, Jupyter Notebook(ipynb), Excel 파일(xlsx), Tableau 워크북(twbx) 등

### 도입한 해결방안

- **JIRA 기반 티켓 관리 시스템**
    - 각 분석 아젠다마다 JIRA 티켓을 발급하고, 관련된 모든 파일들을 해당 티켓에 저장하는 방식 도입
    - 분석 과정과 결과물을 체계적으로 추적 가능
        
- **직관적인 파일 구조 설계**
    - **One-depth 구조 유지:** 복잡한 폴더 구조 대신 한눈에 파악할 수 있도록 평면적으로 구성
    - **파일명 = 아젠다의 답**: 파일 이름 자체가 분석 질문에 대한 명확한 답변이 되도록 명명
        
- 실제 사용하고 있는 파일명 예시
	- [FXDATA-685] 참여 퍼널 개선으로 첫 참여 전환율 5%p 상승.md
	- [FXDATA-703] 이벤트 첫 2주 휴면유저 80만 명 중 20만 명 복귀.md
    
### 추가로 발견한 문제와 해결책
- **해석 차이로 인한 혼선**
    - 동일한 분석 결과에 대해서도 팀원별로 이해도와 관점에 따라 해석이 달라지는 경향 발견
        
- **Data Alignment 미팅 도입**
    - 정기적인 Data Alignment 미팅을 통해,
        - 분석 결과에 대한 해석을 팀 차원에서 정렬
        - 구체적인 액션 아이템 도출 및 공유
        - 데이터 기반 의사결정의 일관성 확보
            
>`COMMENT` 해당 세션에서 가장 인상 깊었던 부분은 단순히 도구나 시스템을 도입한 것이 아니라, 데이터 분석의 전체 라이프사이클을 고민했다는 점입니다. 분석 결과의 저장과 검색, 재활용, 그리고 팀 내 해석의 일관성까지 모두 고려한 흥미로운 접근이었습니다. 팀에서도 바로 적용해볼 수 있을 만큼 실용적이라고 생각합니다.

## 데이터 분석팀 리더는 처음이라

- 성과를 내기 위해서 는 다음 두 가지 필요    
    - **역량(Capability)**: 업무를 수행할 수 있는 실질적인 역량
    - **동기(Motivation)**: 그 일을 하고 싶어하는 마음과 방향성
        
### 역량 개발

- **Self Review → Calibration → Action Plan → Feedback**의 순환 구조로 개인 역량을 체계적으로 발전
- **Self Review**
	- 데이터 분석가에게 필요한 핵심 역량을 10가지로 문서화(학습과 피드백, 오너십, 데이터 분석 방법론 등)
	- 각 팀원이 이 10가지 항목에 대해 스스로를 평가하되, 단순한 점수가 아니라 생각과 근거를 함께 작성
- **Calibration**
	- ‘흠음오와’ 체계 사용 → 흠..., 음., 오? 와! (진짜로)
	- 1on1을 통한 점수 조욜
	- 서로의 평가 점수에 차이가 있을 때, 왜 그런 차이가 생겼는지 대화
- **Action Plan**
	- 부족한 역량 중에서 1~2개 정도만 선택하여 강화하고 보완하기 위한 구체적인 목표와 계획을 수립
- **Feedback**
	- 다음과 같은 질문들을 통해 성장 과정 피드백
		- 목표대로 계획이 진행되고 있는가?
	    - 만약 그렇지 않다면 무엇이 블로커인가? 어떻게 하면 해소할 수 있는가?
	    - 해당 성장 계획을 통해 실질적으로 역량이 성장하고 있다고 느끼는가?
	    - 구체적인 성장 사례들이 있는데, 이것이 실제 성장을 증명하는 사례가 맞는가?

### 동기 부여
#### 팀 목표 설정 및 방향성 정립
- 주요 협업 조직의 미래와 잘 align 되어 있다
- 중장기 목표와 로드맵이 타당하고 매력적이어서, 이루어내고 싶은 마음이 든다
- 시기별 마일스톤과 실행 전략이 충분히 구체적이라 무엇을 해야할지 바로 이해할 수 있다
    
#### 방향성 작성 및 전달
1. **Mission 정의:** 데이터 분석팀이 만들어야 하는 핵심 가치는 무엇일까요?
2. **Goal for next 3 years:** 우리가 만들어야 하는 팀의 미래는 어떤 모습일까요?
3. **Gap Defining:** 현재 모습은 목표와 어떤 차이가 있나요?
4. **Gap Bridging & Milestone:** 이 차이를 해소하기 위해서는 무엇을, 언제까지 해야하나요?
    
>`COMMENT` 이번 발표에서 가장 인상깊었던 부분은 팀의 목표와 (직무적) 역량 개발을 별개가 아닌 연결된 시스템으로 바라본다는 점이었습니다. “A, B, C”와 같은 등급이 아닌 "흠음오와"라는 다소 부드러운 점수 체계도 재밌었고, 평가에서 끝나는 것이 아니라 구체적인 액션 플랜을 같이 세우고, 실행하고, 피드백하는 문화가 인상적이었습니다.

## 중층적, 고객중심적, 동적으로 접근하는 제품 성장
### DEI 프레임워크 개요
- 문제로부터 시작하여 가설을 연속적으로 발전시키고, 지속적으로 인과관계를 증명해 나가는 체계적인 접근법
- **Description <-> Execution <-> Improvement**의 3단계 순환 구조(Undirected)로, 각 단계가 유기적으로 연결되어 점진적인 문제 해결과 학습 가능
    
### 1) Description
첫 번째 단계인 `Description`에서는 문제를 깊이 있게 이해하고 구조화하는 작업 진행합니다.

- **문제 트리 (Problem Tree): 고객 중심의 문제 구조화**
    - 고객의 가장 중요한 핵심 문제를 식별
    - 해당 문제를 풀기 위해 **선행적으로 해결해야 하는 세부 문제들을 계층적으로 나열**
    - 문제들 간의 종속성과 우선순위를 명확하게 구조화
→ 복잡한 문제를 체계적으로 분해하고, 어디서부터 접근해야 할지 방향 잡기

- **이론 (Theory): 고객 행동의 근본 이해**
    - 고객이 왜 제품을 찾고 제품을 이용하는지에 대한 팀의 가설과 생각
    
- **기회-인과 그래프 (Opportunity-Causality Graph)**
    - 문제를 풀기 위해 팀이 실제로 잡아낼 수 있는 구체적인 기회 식별        
    - 기회를 구성하고 있는 다양한 변수들 간의 인과관계를 구조적으로 표현
        
### 2) Execution
두 번째 단계에서는 Description에서 도출한 인사이트를 바탕으로 실제 실행과 검증 진행합니다.

- **사전분석(Pre-Analysis)**
    - 기회-인과 그래프를 통해 도출한 가설을 토대로 사전분석을 진행하여 구체적인 실행안을 기획하는 단계
        
- **결과분석(Result Analysis)**
    - 실행안을 실제로 진행한 후, 그 결과가 어떠한 방향으로 나타났는지 분석
        
### 3) Improvement
- **피드백 (Feedback)**
    - 결과를 토대로,
        - 실행안 자체에 문제가 있었는지
        - 기회-인과 그래프에서 검증이 필요한 요소가 있는지 확인
→ 이런 피드백 과정을 통해 인과를 밝혀 나가고, 가설을 연속적으로 발전시킵니다.

> `COMMENT` 연사가 세션 시간 내에 프레임워크와 실행 사례를 모두 다루기 어렵다고 언급했을 만큼 진행이 빠르게 이루어져서, 제가 올바르게 이해했는지 모르겠습니다.
> 
> 일단 가장 인상적이었던 부분은 가설을 연속적으로 발전시키는 접근법이었습니다. 문제 해결을 한 번의 실행으로 끝내는 것이 아니라, 우선 체계적으로 정리하고, 기회-인과 그래프의 다양한 가설을 사이클을 통해 검증하고, 그 결과에서 얻은 인사이트를 유기적으로 연결해나가는 방식이 흥미로웠습니다.
> 
> 실제로는 변수 간에 인과관계가 존재하지만 그 영향력이 약하거나, 인과관계는 강하지만 가드레일 지표를 떨어뜨리는 교란 변수가 있는 등 복잡한 상황들을 지속적으로 분석과 실험으로 풀어나가는 사례들을 보고, 당장 전부를 이해하진 못했지만 배우고 실행해볼 부분들이 많다고 생각했습니다.

## 토스가 빠르게 성장할 수 있는 이유 with TUBA

- 빠른 실행을 위한 필수 요소    
    - **실행 속도**: 낮은 실행비용 유지
    - **실험 빈도**: 누구나 쉽게 실험 설계
    - **학습 효율**: 빠른 실험 결과 도출, 성공 경험 자산화
        
- 이를 위해 **TUBA(Toss User Behavior Analyzer)** 개발
    - 초기에는 어려운 실험을 누구나 쉽게 설계하고 실행할 수 있도록 하기위해 시작
    - 지금은 메시징, 실험, 타겟팅, 분석 등 다양한 기능 제공

- TUBA 기능
	- 실험을 설정하면 A/B테스트 결과를 손쉽게 볼 수 있음
	- 이미 구성되어 있는 약 10만개의 세그먼트가 있으며, 직접 선택하거나 ML/LLM을 통해 추천받아 사용 가능
	- 실험 히스토리를  누가/언제든지 확인 가능
	- A/B 테스트 결과를 LLM으로 해석해주는 기능 탑재


> `COMMENT` 기능적 측면에서는 저희도 타겟팅과 메시징을 제공하는 서비스를 가지고 있는데요, 이 서비스에 실험과 분석 기능을 어떻게 추가할지, 그리고 기존 기능들을 어떻게 고도화할지에 대한 다양한 가능성을 확인할 수 있어서 매우 유익한 기회였습니다. 


TUBA 세션에서 나온 말이 인상적이어서 해당 말을 마지막으로 글을 마치겠습니다 :) 


> 제품 개발에서 모든 것을 확신할 수는 없습니다.  
> 하지만 확실하게 말할 수 있는 것이 있다면, 바로
> 
> **의견보다는 실험(Experiment over opinion)**
> 
> 제품을 개발하면서 우리는 종종 고객이 무엇을 원할지, 어떤 기능이 잘 작동할지를 미리 예측하려 합니다.  
> 물론 가설을 세우는 건 중요하지만, 가장 확실한 방법은 실제로 실험해보는 것입니다.
> 
> 짧은 토론으로 추측을 반복하기보다는, 작게라도 실험을 진행해 데이터를 통해 판단하는 것이 훨씬 생산적입니다. 빠르게 아이디어를 검증할 수 있도록 A/B 테스트 플랫폼(TUBA), 배포 시스템 등을 내부적으로 개발해, 누구든 쉽게 실행할 수 있도록 했습니다.
> 
> 어떤 것이 맞는지 말로 따지기보다, 실험을 통해 검증하는 문화.
> 
> 그것이 우리가 더 나은 제품을 만들 수 있었던 이유 중 하나입니다.
